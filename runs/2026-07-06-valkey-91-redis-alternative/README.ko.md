# Valkey 9.1 vs Redis 8 — 성능이 정말 갈아탈 이유가 되나

📝 글: [KO](https://var.gg/ko/blog/valkey-91-redis-alternative) *(한국어만 발행)*
🗓 실행: 2026-07-05 · 🤖 실행 주체: **agent** · 👤 운영자: curioustore
🌐 English: [README.md](./README.md)

> **⚠️ 하네스는 백필이다.** 아래 수치는 2026-07-05 세션 노트에서 **그대로 옮긴 실측값**이지만,
> 당시엔 **하네스 파일이 없었다** — `redis-benchmark`를 셸에서 즉석으로 돌리고 그 명령줄을
> 산문으로 기록해 뒀을 뿐이다. `run.sh`는 그 기록된 명령들을 **재구성**한 것으로 2026-07-23에
> 작성했다. 동작과 메모리 결론을 확인하려고 1/100 규모로 실제 실행했다(아래 *재검증*).
> `manifest.json` → `backfill_note` 참조.

## 주장 ↔ 근거

단일 호스트(Windows 11 + Docker/WSL2), 두 엔진 모두 로컬 컨테이너, standalone, 기본 설정.
이미지는 태그가 아니라 **digest로 고정**했다.

| 글의 주장 | 근거 | 값 |
|---|---|---|
| 같은 데이터셋을 Valkey가 약 23% 적은 메모리로 담는다 | `results.json` → `metrics[memory_footprint_same_dataset]` | **67.86 MB** @ 632,539키 vs **88.31 MB** @ 632,438키 → **-23.2%** (Redis +30.2%) |
| …키 수가 실제로 맞춰진 상태에서 | 같은 표 | 약 632.5천 개 중 차이 **101개**(0.016%) |
| 파이프라인 없는 단건에선 Valkey가 앞선다 | `results.json` → `metrics[throughput_unpipelined]` | SET 315,457 vs 277,778 rps · GET 296,736 vs 270,270 rps (**~9–13%**) |
| 파이프라인을 깊게 걸면 Redis가 앞선다 | `results.json` → `metrics[throughput_pipelined]` | P=16: SET 2.27M vs 2.06M · GET 2.70M vs 2.41M (**~10%**, 방향 반전) |
| 그래서 "성능 때문에 갈아탄다"는 약한 논리다 | `results.json` → `findings[performance_is_a_weak_migration_argument]` | 우열이 **워크로드 모양에 따라 뒤집힌다**. 메모리 격차는 뒤집히지 않는다 |

## 재검증 (2026-07-23)

재구성한 하네스가 실제로 도는지, digest 고정 상태에서 **클라이언트 하나로 두 서버를 모두 때렸을 때**도
메모리 결론이 유지되는지 보려고 `SMOKE=1 ./run.sh`(1/100 규모)를 같은 호스트에서 두 번 돌렸다.

| | 키 수 | used_memory | 키당 바이트 |
|---|---:|---:|---:|
| Valkey 9.1.0 (1회 / 2회) | 6,273 / 6,358 | 1,754,296 B / 1,761,880 B | 279.7 / 277.1 |
| Redis 8.8.0 (1회 / 2회) | 6,295 / 6,317 | 2,352,104 B / 2,354,344 B | 373.6 / 372.7 |

→ 키당 **25.1% / 25.7%** 적다(전체 규모에선 23.2%). 서버 버전도 확인됐다
(`valkey_version:9.1.0`, `redis_version:8.8.0`, 둘 다 jemalloc-5.3.0).
원본 출력: [`smoke-2026-07-23.txt`](./smoke-2026-07-23.txt).

**스모크 모드는 처리량을 판정하지 못한다** — 요청 5천~1만 건에선 `redis-benchmark`의 rps 출력이
거칠어 두 엔진이 같은 구간에 떨어졌다. 처리량은 전체 규모 실행만이 말할 수 있다.

## 정직하게, 검증하지 않은 것 / 한계

- **단일 호스트 마이크로벤치마크.** production 토폴로지가 아니다. p50/p99 분포는 보존하지 않았고
  처리량만 남겼다.
- **거의 전부 n=1.** 반복한 건 파이프라인 없는 SET 셀 하나뿐이고, 그 한 번의 반복에서 이미 Redis SET이
  약 4% 움직였다(277,778 → 289,855). 단발인 파이프라인 ~10% 차이가 이 런에서 가장 약한 숫자다.
- **원래 실행에서 엔진별로 어떤 클라이언트 바이너리를 썼는지 기록이 없다.** `run.sh`는 redis:8의
  `redis-benchmark`를 양쪽에 써서 그 변수를 제거했다 — 실험으로는 더 낫지만, 재현 시 기록된 rps에
  정확히 안 떨어질 수 있다는 뜻이기도 하다.
- **각 단계가 갓 띄운 서버에서 시작했는지도 원본 기록에 없다.** `run.sh`는 엔진마다 새 컨테이너를
  띄우고 단계 사이에 `FLUSHALL`한다.
- **클러스터·복제·영속화 없음.** 글에서 다룬 Valkey 9.1의 cluster bus / failover 변경사항은 하나도
  측정하지 않았다.
- **신규 명령**(`HGETDEL`, `MSETEX`, `CLUSTERSCAN`)과 cluster bus 트래픽 지표는 **릴리스 노트 인용이지
  측정이 아니다**.
- **Windows + Docker Desktop(WSL2 백엔드).** 네이티브 리눅스에선 두 엔진의 절대 rps가 달라진다.
  넘겨받을 수 있는 건 엔진 간 비교뿐이다.
- Valkey의 `INFO`는 `valkey_version:9.1.0` 옆에 `redis_version:7.2.4`를 같이 보고한다. RESP 호환
  표식일 뿐 서버 세대가 아니다. "Valkey는 Redis 7.2"로 읽으면 안 된다.

## 재현

```bash
./run.sh            # 전체: 5~10분, 대부분 100만 키 메모리 단계
SMOKE=1 ./run.sh    # 1/100 규모 동작 확인, 약 1분
```

비슷한 `DBSIZE`에서 Valkey가 메모리 **20~25% 적게**(키 수가 1% 넘게 벌어지면 총량 말고 키당으로 비교),
단건에선 Valkey가, 파이프라인 깊이 16에선 Redis가 앞서면 재현된 것이다.
**핵심은 부호가 뒤집힌다는 사실이고, 퍼센트는 노이즈다.**

## 환경

Windows 11 + Docker 29.4.3, WSL2 백엔드(Linux 6.6.114-microsoft-standard-WSL2, x86_64) ·
`valkey/valkey@sha256:4963247a…`(서버 9.1.0) · `redis@sha256:2838d552…`(서버 8.8.0) ·
둘 다 jemalloc-5.3.0, 64-bit. 두 digest 모두 2026-07-23에 Docker Hub에서 여전히 해석되는 것을
확인했다. **핀은 digest다** — `9.1`·`8` 태그는 움직인다.
